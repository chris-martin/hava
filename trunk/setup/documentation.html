Project Vision


Overview

Hava is a language created to help solve problems in industrial and systems engineering.  The language is meant to be used as a learning tool for aspiring engineers who might not be familiar with typical programming languages or paradigms.

The purpose of this project is to create a development environment for Hava.  One Hava editing tool currently exists, but needs to be expanded upon.  The primary requested feature is the inclusion of debugging features.

This feature is part of a larger vision to create a framework which will promote usage of the Hava language.  Hava is a new way to solve a domain problem, and its creators would like to see it popularized.  An important aspect of this goal is to make it attractive to non-programmers by keeping initial learning overhead small.

Problem Statement

The problem
 Complicated Hava code is difficult to debug
Who the problem affects
 Anyone who uses the Hava language
The impact of the problem
 Time is wasted, Hava programs used in classes or real applications may be defective
A successful solution
 A system which clearly shows the runtime execution of Hava code, particularly at a point of failure.

Product Position Statement

Who the users are
 Experts in engineering fields, with little programming knowledge
What the users do
 Use Hava to solve dynamic programming problems
What our system is
 An Eclipse plugin with tools for authoring, navigating, and debugging Hava code
What our system does
 Displays the details of Hava code execution, in an interface which is linked to the code editor
Existing solution
 A basic text field for editing, with plain text output
Advantage of our system
 A clearer and more interactive interface makes it easier to understand how Hava code is being executed.

User Demographics

The sponsors anticipate two distinct categories of Hava users within the industrial engineering community.

  • "Power users" consist of graduate students, instructors, and industry professionals.  The major feature additions such as improved debugging are targeted mainly toward these users.  Power users may or may not have previous programming or IDE experience outside of the existing Hava editor.  The power users are used to the current IDE, so the new one should be designed to look and feel like the current IDE.
  • The majority of users- around 80%- will be upper-level undergraduate students who do not need to write or debug their own Hava code.  Instead, they will be running and making minor modifications to existing Hava programs written by a power user.  These users are likely to have never used Hava or an IDE before.  They have no previous experience which would allow a good basis for UI design, other than to make it simple and accessible.

The main purpose of our product is to assist the power users, we also want to keep in mind that others may have reason to use it as well.

Features

Hava source code editor
  • Syntax coloring
  • Parse checking - syntax errors will be indicated in files being edited.
  • Tabbed view for editing multiple file
  • Hava files can be opened from a URL

Launching Hava code
  • Hava files can be run in launch or debug mode, opening the appropriate view
  • Time elapsed and memory usage will be displayed for every Hava file launched

Hava solution view
  • A view containing the Hava report: a formatted table of variable values and comment
  • Variables and comments can be sorted alphabetically and filtered by a text search
  • Includes a method to jump to relevant Hava code

Hava debug view
  • A tree view to be used for debugging.  Each variable is given a node which may be expanded to show how that variable was calculated.  This view will replace the current "trace" feature.
  • Includes a method to jump to Hava code referred by a particular node
  • Variables can be sorted alphabetically and filtered by name

Cross Platform Support
  • Ability to be installed on different platforms.
  • Pre-built installation for Windows (just extract a zip file and run an executable).

Constraints

  • Accessibility - The IDE, and the debugging functionality especially, should give insight into how the language works so that users are better able to learn the language.
  • Debuggability - When a user makes a mistake in their code, finding the error and correcting it should be an easy task.
  • Usability - Running and debugging should be simple and intuitive.
  • Simplicity - Many end users have never developed before so an IDE is not second nature to them.  The simpler and more straightforward it is to do things, the easier they will be able to get their work done.
  • Flexibility of Environment - Experienced developers and neophyte should both be able to use the IDE.
  • Maintainability - Future developers may need to fix bugs in the IDE.  Doing so should be simple.
  • Extensibility - Any new features that are needed should be able to be added to the IDE easily.

Eclipse Implementation

Plug-ins can be bundled into an "Eclipse product"
  • Download eclipse and Hava plug-in at the same time
  • Stripped of unnecessary features
  • Carries Hava "branding" (such as replaced splash image)

There are two options for how we deploy the application
  • Simple option: Install JRE, download and unzip the eclipse product.
  • More sophisticated: We can create installers for Windows/Mac/Linux, which will install the JRE if necessary and set up file associations for .hava files.

Addressing ease of use
  • The Hava perspective automatically opens the commonly used views (navigator, Hava editor, Hava solution)
  • The product ships with a workspace containing a "Default Hava Project", so users disinterested in creating projects can deal only with files in this project if they prefer.
  • Default Hava Project comes with an immediately working example of Hava code, which demonstrates both the power of the language and the functionality of the eclipse perspective.

Additional benefits of Eclipse
  • Many basic features like tabbed editing and text search already exist.
  • Eclipse is a mature platform, and our Hava features will be extensions to existing stable, polished components.
  • Software update manager allows us to provide plug-in updates easily and uniformly.

Applet / Standalone Implementation

Without the underlying architecture that Eclipse provides, developing matching features for the existing editor will be more difficult. The applet is a beneficial product because it can be used with no installation whatsoever.  However, the addition of any features here is outside the scope of this project.

We must make sure that our changes to the Hava engine do not break any functionality in these applications, and the applet must continue to be compliant with Java 1.5.

Detailed Design

Package Structure

At the highest level, the design is separated into three packages:

  • Hava Common contains code which is not particular to any specific application or user interface, including the actual Hava Engine which compiles and runs Hava code.  The bulk of this package is the Hava language implementation (Hava Core) written by Loren Platzman.  Some modifications have been made to accomplish debugging features; the Debug subpackage contains the hooks into the Hava engine to retrieve debugging information.
  • The Application package contains a primitive IDE as both a standalone app in the Standalone package, or a web applet in the Applet package.  It was also written by Loren Platzman.  Originally the Hava Common and Hava App packages were closer, but to separate responsibilities and make package dependancies cleaner, they were divided.  Hava Gui contains the main Java Swing component of the original IDE.  Hava Applet and Hava Standalone wrap this component to make it an applet or standalone gui application, respectively.
  • The largest part of the new code belongs in Hava Eclipse, which contains all of the Eclipse-specific code for the plugin.  Following Eclipse convention, this is divided into many smaller packages, each containing a single plugin.  Hava Feature aggregates all of the other plugins so they can be installed as a single entity.




Eclipse Features

Builder

The Builder plugin performs a continuous build of Hava files in the workspace, re-parsing whenever a file is modified.  Its purpose for Hava is to detect parse-time errors, generating error markers which are used by other plugins to display error messages in other parts of the UI.  This type of extension in Eclipse also is typically used to create persistent resources (such as compiled Java class files), but this is not relevant to Hava.

Launch

The Launch plugin is used for running Hava files.  Its most important responsibility is the launch configuration, which executes Hava code and sends the result to the views.  It also hooks the launch configuration into the Eclipse user interface by adding toolbar and menu buttons which allow the user to request the execution.

Editor

The Editor plugin is is where the UI for document editing lives.  It scans and parses Hava documents to apply coloring.

Perspective

The Perspective plugin contains configuration for the Hava perspective, including its name and icon attributes handles.  It contains a perspective factory which initializes the default workbench layout with the views most essential to Hava development.

View

The View plugin holds the UI that displays the output of a Hava run, as well as the debugger for the same run.

Common

The Common plugin is a place for any code which is specific to both Hava and Eclipse, but does not fit in any other plugin.




Dynamic Design

Interactions Between Eclipse Plugins

The Eclipse IDE has a plug-in architecture that handles common IDE functionality.  It uses an event-driven design to pass information to the different parts of the IDE.  This is done through extension points.  Each extension point can contact other parts of the plugin or other plugins through an IContactRegistry.


Coupling between plugins is generally very low.  For instance, the Perspective plugin initially activates two Hava views, but the View plugin is not in Perspective's classpath - instead of directly dealing with the views, it requests them from the IDE, referencing them by a string identifier.

Continuous Parsing

The Builder and Editor plugins both parse Hava files as they are modified.

When any file in the workspace changes, Eclipse fires an event which is recieved by the project's builders. Hava projects have a HavaBuilder. When the builder catches this event, it checks if enough text has changed to warrant a rebuild. As an IncrementalBuilder, it has the option to try to recompile just the regions that have changed. If this is not feasible, it will rebuild the entire file as well as any dependencies within the project. These rebuilds are done through the Hava Engine provided by the Common Package. If any problems are found in the syntax, HavaBuilder creates an error IMarker and attaches it to the IFile resource. Adding this marker causes the Eclipse framework to add error annotations to the Project Explorer view, Problems view, and Hava editor (if the file is currently open).

When a file is modified inside the Hava editor, HavaEditor's DocumentProvider breaks it into regions. The DocumentProvider then fires an event to Eclipse notifying it that the regions have changed. The HavaEditor catches this event and recolors each region and checks it for errors again using the PresentationReconciler.

Debug Listeners

As the HEngine executes, it fires events to provide information to debugging tools.  A class which listens on debug events must implement the HDebugListener interface, and an instance must be hooked up to the engine by calling HEngine#addDebugListener(HDebugListener) to begin receiving events.

Launching

When the user requests that a Hava file be run, Eclipse calls the launch method of HavaLaunchConfigurationDelegate, providing the following information:
  • The ILaunchConfiguration (knows which file to run)
  • The launch mode string ("run" or "debug")
  • An IProgressMonitor (used to detect if the user cancels the launch)

This method instantiates a LaunchTask to handle running the Hava code.  Based on the requested launch mode, this is either a RunTask or a DebugTask.  The difference between a "run" and a "debug" launch is the amount of information collected, so these differ only in which HDebugListeners are applied.

After running the Hava engine, the results are sent to the view.  Because the launch code is not executed in a UI thread, it cannot interact directly with the workbench.  Instead, it creates a Runnable called LaunchUITask to update the view. This is passed as an argument to Display.getDefault().asyncExec(Runnable), and Eclipse's UI thread runs it.

Aborting a Launch

To cause an HEngine to abort its execution, an HEngine can be given one or more HProgressMonitors.  An HProgressMonitor has a boolean checkAbort() method, so that as the engine runs, it can periodically check whether any of its progress monitors have requested an abort.  There are two events that can trigger an abort: if the user specifically requests it, or if system memory is running low.

The Eclipse framework provides a mechanism for cancelling a launch.  The return value of IProgressMonitor's boolean isCanceled() method indicates whether a cancel has been requested.  The IProgressMonitor is trivially wrapped in an HProgressMonitor called HavaMonitor.

Sometimes a Hava script needs to be aborted because it requires more than the available amount of memory.  As memory usage approaches this limit, application performance degrades unacceptably.  To get around this problem, another HProgressMonitor called MemoryMonitor is added.  MemoryMonitor polls java.lang.Runtime to approximate the percentage of used memory and abort if it exceeds some threshold.  Instead of running in a separate thread, MemoryMonitor is implemented as an HDebugListener, and its poll interval is defined as some number of debug events.

Swing Applications

Both the standalone application and the applet are centered around the HPanel class, which constructs and controls the rest of the components.  It runs two threads, referred to as the monitor and work threads.
  • The monitor thread runs continually, probing the input JTextArea to detect changes.
  • When a change in the JTextArea occurs, the monitor thread sends the Hava code to the work thread.
  • If the work thread is already active, it halts its current execution.
  • It then starts to compile and run the new code using the Hava engine in the Common package.  If the Trace checkbox is checked, it first adds a TraceDebugListener to the engine.
  • When the work thread finishes running the Hava script, it writes the resulting report (or the Trace output) into the HPanel's output JTextArea.

Data Design


For debugging purposes, a tree of variable names and values is built while the source is being compiled.  Each node represents a variable or a statement, and its children are the variables it depends on.  The root nodes are variables that no other variables depend on.  This structure is only maintained in memory, and never gets written to disk.

The sole persistent data source in the Hava IDE are the hava source files.  They are text files with a .hava extension and contain the source code that the IDE compiles and runs.

The full format specification for the source code can be found here:

www2.isye.gatech.edu/people/faculty/Steven_Hackman/Hava/HavaTechnicalReference.pdf

SAAM Analysis

Future Features

Code Completion

To implement code completion- called content assist in the Eclipse framework- the HavaSourceViewerConfiguration, found in the hdt.editor package, needs to be edited, and implementations of IContentAssistant and IContentAssistProcessors need to be placed in the same package. hdt.editor also needs an appropriate action or command- depending on which API is used- implemented to attach a keyboard shortcut or button to the editor interface.

Code Generation

To add code generation, a few things need to be done. An easily traversable document object model needs to be created for each Hava source file every time the file is built, affecting the hdt.builder package. This DOM needs to be stored in the workspace as a resource of the current project. An action or command implementing the code generations needs to be created in hdt.builder, and associated with Hava source files by the HavaNature.

Opening Hava Source from a URL

First, IStorageInput needs to be implemented in the hdt.editor package to allow read-only source from a URL to act as an editor input. Then, the jump-to-declaration action in hdt.views.solution.tree needs to recognize URL imports, and open them as the new URL implementation of IStorageInput, instead of the usual IEditorInput.


Note:  This feature was implemented in the third sprint.  The above analysis was correct, except that a more general Eclipse command was subclassed for re-use by other plugins.

Summary

The traceability analysis was revealing. Most of our features were covered by one to two packages, suggesting that features were organized modularly. The analysis also revealed that we have yet to work out the Eclipse help system, or design a package in our plug-in to provide for help.


The plausible future features in the SAAM analysis suggest that our architecture was fairly well chosen. Each feature primarily affects one package; other packages are affected only because of user interface components whose location is mandated by Eclipse standards.


Design Rationale


The central architectural question involved in our senior design project was whether to primarily extend the existing Hava editor, or use the Eclipse framework to create a newer, more powerful Hava IDE. The two directions differ significantly both in view of architecture and our client's requirements.

Solution Comparison

Standalone Enhancement

Description

The existing Hava editor, a Swing application with basic file I/O capabilities, an editing pane, and an output/trace pane, offered a base line for the project, and an opportunity for extension.

Advantages

Architecture tailored to our solution

The architecture behind improvements to the current Hava editor has the advantage of being lean and mean. After a large refactor of the existing user iterface, design will be practically ground up; the new editor will not have a single architectural feature for which it does not have an obvious need, in contrast to the hoop-jumping frameworks sometimes require.

Better control over user interface

Enhancing the current editor allows us free reign to design an interface to best work for the projected typical Hava user.

Disadvantages

Difficult given time constraints

The scope of the project will be difficult to fulfill, from the ground up, with the summer's time constraints.

Opposition to code base

The current code base is not necessarily a great launching point for some of the client's requirements. The user interface code was not built to be extended, and many of the features required of an editor now were not foreseen at the editor's inception.

Eclipse Integration

Description

Eclipse is an plug-in based IDE that has evolved into a full application framework. Not only are there plug-ins for development in Java, its native language, and many other higher level languages, but also a slew of software design and development tools, and even tools for managing projects at a level much higher than a traditional IDE. The flexibility of the platform increases the complexity; integrating a language fully with Eclipse requires a serious effort.

Advantages

Existing code base

Many of the features driving our project have already been implemented in the existing Eclipse code base. Multi-file and tabbed editing, frameworks for syntax highlighting, in-editor help, and debugging can all be found in Eclipse. Eclipse also defines user interface standards so that plug-ins work similarly to one another and that the entire IDE remains consistent to the user.

Modularity

The Eclipse plug-in registry enforces design modularity. Different functional pieces of an Eclipse plug-in are not allowed to interact, except through a static registry lookup, seriously reducing coupling. For this same reason, plug-in components are easily swapped.

Maintainability

Eclipse already claims a large developer base. Future Hava maintainers familiar with Eclipse will be given an edge, and will also benefit from extensive Eclipse documentation to supplement that of our project.

Extensibility

Plug-in fragments allow easy plug-in extension in the Eclipse framework. The plug-in registry also makes is possible for future plug-ins to easily cooperate with the Hava IDE.

Popularity

Hava's integration with Eclipse, an already popular platform, might help popularize the Hava language itself.

Disadvantages

Weight

The Eclipse framework might prove to be too cumbersome for a project with the size and time constraints of our own.

Models of Use

Certain interface paradigms, such as workspaces, projects, an auto-bulding IDE, and run configurations, are foreign to the project typical Hava IDE user. These would need to be hidden or well explained to keep the IDE accessible to those that aren't used to modern IDEs.

Summary

Our client's requirements are a roughly prioritized list of IDE features, including better debuggability and syntax coloring without the sacrifice of existing features, accessibility for new users of the Hava language, and other potential features focused on increasing the popularity of the Hava language and creating a more effective work-flow for power users of the language. The existing Hava editor offered a relatively clean canvas for the project. Syntax coloring would be simple to implement in such an environment, but other IDE features, like tabbed editing, syntax checking, and, most importantly, a strong debugging environment, would be very difficult. On the other hand, the Eclipse framework is used often for these very features. Integrating Hava with the Eclipse framework allows a much richer feature set than writing the editor nearly from scratch, and also offers a more professional tool that is built to be maintained and extended.

Solution

We decided, based on our features requirements and constraints, to focus on Eclipse integration, and to add features to the existing editor when there was enough overlap to justify doing so.

The choice to use such an extensive framework is in strong architectural contrast to the option of building on the existing editor.

The Eclipse framework's architectural rigor is obvious. It supports a vast number of different plug-ins, and is used as a building block for many other large applications that are only tangentially related to the IDE.  For this  reason, designing for the Eclipse framework is less an act of creating a strong architecture than it is a study of existing methods.  Many of the detailed design choices were removed or made simpler; instead, our design phase has consisted of synthesis of documentation into a working model of the Eclipse framework, and our plug-in's relation to that framework.


Test Plan


The product is an IDE for the Hava language, for which a parser and interpeter has already been written.  The majority of our new code consists of add-ons to the Eclipse application, but we also need to maintain the quality of the Hava core interpreter.

Strategy

Roles

Black-box Testing

Most testing deals mostly with the user interface.  Our code is more about connecting and configuring third-party components, so it does not contain a great deal of unit-testable logic.   Due to the temporal limits of this project, automated UI testing is not worth the initial setup overhead.  UI components are tested at the end of each week using the manual walkthrough UI tests enumerated below.

White-box Testing

The biggest potential source of logical errors is the Hava interpreter.  This component ought to be unit tested heavily (and isn't), but is not our responsibility because we do not have significant ownership over it.  However, some amount of testing may be appropriate for the minimal changes we have made to support our new functionality, particularly relating to the interpreter's new debugging interface.  JUnit is used where appropriate.

System Testing

The Eclipse projects are platform-agnostic, so only the installation packages will need to be tested on each supported platform.  Windows Vista is the only platform the sponsors care to support, however, so testing on multiple systems is not worthwhile.

The primary mode for validation testing will by obtaining direct feedback from Steve Hackman and Loren Platzman, the project sponsors.  To identify UI aspects are unclear or cumbersome, feedback about general usability concerns will be obtained through a combination of open-ended questioning and structured UI walkthrough.

Basic Usability

The previous version of the IDE can serve as a basis for comparison, because it is currently in use effectively.  The system can be considered to satisfy this category of testing if it equals or exceeds the usability of those features which are implemented in its simpler predecessor.

File Management

Eclipse's project structure necessitates some differences in how files are managed.  At minimum, without delving into the mechanics of Eclipse's concept of "project" and "workspace", we need to verify that the environment intuitively supports file management.

  • Create new Hava files

Running Hava Files

The current IDE has a single text field, and an output panel which can display the result ("solution") or debugging information ("trace") for the Hava file presently undergoing edit.  Because the Eclipse editor can open multiple Hava files at once, its running is not as straightforward.  Since running Hava code is the most important task the system performs, validating the run interface is critical.

  • Run a Hava file (open the Solution view)
  • Debug a Hava file (open the Debug view)
  • Running a specific Hava file when multiple editors are open

Debugging Features

The purpose of the new functionality introduced by this Hava IDE is to facilitate debugging.  To determine whether this goal has been satisfied, we have test situations which address specific instances where better debugging assistance is desired.  A successful resolution to each of these scenarios means that the Debug view made the problem apparent and assisted in its solution.

Poker Example: Type Mismatch

This example was provided by Loren Platzman.  The following faulty Hava code fails on a type mismatch error:

token JACK, QUEEN, KING, ACE;
FACES = (2, 3, 4, 5, 6, 7, 8, 9, 10, JACK, QUEEN, KING, ACE);

token CLUB, DIAMOND, HEART, SPADE;
SUITS = (CLUB, DIAMOND, HEART, SPADE);

struct Card(face, suit);

function randomCard = Card(random(FACES), random(SUITS));

averageValue = sum(i=1 to 100) {randomCard.face} / 100;

This code fails during the summation when randomCard.face is non-numeric.  Because the error occurs on an anonymous variable (the return value of the randomCard function), the trace is difficult to read because it is unclear about which value was problematic.
